The Content Widget allows you to create re-usable HTML code snippets that can be included throughout your application. The example below shows a tab panel (see Tab Panel for more information) that displays different HTML files based on the active tab. When you click a tab, it triggers the display of an external HTML file specified in the Content Widgets definition. You can also pass arguments to a content file to do things like set page titles (like in the example below) or anything else you can think up.

The content of the Content Widget should be a well-formed XHTML page. This is not a requirement, just a good thing to do. Having your content in a valid XHTML page means that they can be used independently, if required, and indexed by search engines, such as Google. The Content Widget applies any styles defined in the header and removes conflicting markup, such as <html> tags.

The way Content Widgets work can expose a few quirks. The Content Widget does nothing with duplicate id's. Also, the content of a Content Widget is loaded initially in a hidden iframe then loaded into the DOM of the parent page. Although the Content Widget will eventually throw away any script tags defined in the header, they will be evaluated when the content is parsed in the iframe. This means that including references to large javascript files (such as Appcelerator) will cause performance degradation. Any javascript that the include file will need should either be included in the parent file or included in an <app:script> block.

Required Options:

on the expression that determines what message will trigger the execution of the Content Widget. The correct syntax is :on="[message_name] then execute"
src URI for the content source (should reference a HTML file)
lazy whether or not to lazy load the content file (true|false)

Optional Options:

args JSON representation of parameters. For example: args="{'id':'123'}"
onload The name of the message to be published when the content is loaded (content lifecycle is fetch, compile, load, then execute). This will carry a payload of src and args so that you can uniquely identify the callback.
onfetch The name of the message to be published when the content is fetched (content lifecycle is fetch, compile, load, then execute). This will carry a payload of src and args so that you can uniquely identify the callback.
